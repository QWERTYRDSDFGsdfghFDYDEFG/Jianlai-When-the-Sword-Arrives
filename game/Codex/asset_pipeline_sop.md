# 从立绘创建到场景构图再到 Ren'Py 落地的完整 SOP

适用范围：`game/` 全工程。

用途：把“角色出图、场景构图、Ren'Py 接入、文档同步”串成一条固定流程，减少反复返工、命名混乱、角色漂移、场次画错和脚本接不上的问题。

---

## 0. 一句话总原则

先锁定“这个人是谁”，再锁定“这场戏怎么演”，最后才落到“这张图怎么接进 Ren'Py”。

本项目默认执行顺序：

1. 先读规则与索引
2. 再判定资产类型
3. 再做角色或场次出图
4. 再导入资源
5. 再接 `bg_images.rpy` / 脚本 / 配音
6. 最后做静态检查与验收

---

## 1. 开工前必须先读什么

### 1.1 通用入口

按顺序先读：

1. `README.md`
2. `GDD.md`
3. `Codex/00_MASTER_PROMPT.md`
4. `Codex/01_FIRST_PLAYABLE.md`
5. `Codex/story_compression_record.md`

### 1.2 做角色图前必须读什么

无论是立绘、表情图、动作图、场景图还是某场戏的事件图，先读：

1. `Codex/character_image_index.md`
2. 该角色对应的专属身份标准文件
3. `角色立绘设计原则.md`
4. `《剑来》VN演出与构图总手册.md`

如果是“某角色在某场戏里的图”，还必须继续读：

1. 对应章节的逐场分镜表
2. 对应章节的 Ren'Py 演出脚本执行表

当前默认回查规则：

- 序章 / `script.rpy`：`script逐场分镜表.md` + `script_RenPy演出脚本执行表.md`
- Chapter 2-5：`script_chapter2~5逐场分镜表.md` + `script_chapter2~5_RenPy演出脚本执行表.md`

结论：

- 角色身份标准负责“这个人是谁”
- 分镜表负责“这场戏要什么画面”
- 执行表负责“这张图怎么在 Ren'Py 里落地”

## 2. 先判定这次要做的到底是什么图

不要一收到需求就直接写提示词。

先判断这次需求属于哪一类：

### 2.1 基础立绘

用途：

- 日常对白
- 常规左中右站位
- 同一角色长期复用

回答的问题：

`这个人平时怎么站着说话`

### 2.2 动作立绘

用途：

- 停步
- 侧听
- 接话
- 追问
- 退让
- 并行行走

回答的问题：

`这个人此刻怎样承受、观察、开口`

### 2.3 事件构图图

用途：

- 登场
- 奇观
- 武戏
- 群像压迫
- 必须看见空间关系的强事件

回答的问题：

`这场戏正在发生什么`

原则：

- 不能把事件图降级成普通站姿立绘
- 不能把普通立绘硬抬成事件图

## 3. 角色资产的标准生产顺序

不要一上来就做“新表情 + 新动作 + 新镜头 + 新场景”。

本项目固定按下面顺序积累角色资产：

1. 脸部锁定近景母版
2. 通用全身站姿母版
3. 同镜头表情差分
4. 小动作差分
5. 中动作差分
6. 特定场景演出图

### 3.1 第一阶段：先锁身份

目标：确认还是同一个人。

要求：

- 优先中近景或半身
- 正常光线
- 中性或轻表情
- 不做大动作

若这一阶段不稳，后面所有变体都会漂。

### 3.2 第二阶段：再做表情差分

目标：在不换人的前提下，让表情真的不同。

要求：

- 镜头先尽量不变
- 动作先尽量不变
- 一次只改一个情绪方向

表情不要只写抽象标签，要拆成可见变量：

1. 眉
2. 眼睑 / 眼神
3. 嘴角 / 唇形
4. 面颊 / 下颌张力
5. 头部微角度

### 3.3 第三阶段：最后做动作差分

目标：在身份稳定、表情稳定后，再让身体活起来。

动作不要只写名词，要至少拆成：

1. 身体朝向
2. 重心在哪条腿
3. 另一条腿在做什么
4. 骨盆是否偏转
5. 肩线是否倾斜
6. 左手在做什么
7. 右手在做什么
8. 头朝向哪里
9. 衣摆 / 发尾是否跟随

硬规则：

- 一张图只允许一个主动作
- 有动作就必须有重心
- 支撑腿必须稳，自由腿必须有任务

## 4. 场次构图的标准生产顺序

### 4.1 先拆场，不先出图

先把剧本拆成“可制作场次”，每场至少写清：

- 剧情节点
- 背景
- 图像类型
- 动作级别
- 出场角色
- 表情
- 站位
- 动作设计
- 禁区
- 景别
- 演出备注

本项目已经使用的动作级别：

- `A0` 标准静立
- `A1` 轻动作对白
- `A2` 关系动作
- `A3` 事件动作

执行规则：

- `A0/A1` 优先考虑基础立绘或轻动作立绘
- `A2` 必须写清角色关系动作，防止退化成普通站姿
- `A3` 优先考虑整图事件构图，不靠普通立绘硬演

### 4.2 再决定这场是“母图”还是“派生图”

不要默认每场一张新图。

优先判断：

1. 这场能否复用已有环境母图
2. 这场是否只是同一空间下的景别变化
3. 这场是否只需要角色换站位、换表情、换压暗
4. 这场是否真需要新事件图

推荐资产结构：

- 主母图：交代空间和世界观
- 派生图：同一空间下的角度、景别、情绪变化
- 纯切分图：同一图上靠 Ren'Py 立绘调度推进对白

### 4.3 景别选择规则

- 远景：开场、转场、环境先行、人小景大
- 中景：项目主力景别，承载大多数对白
- 近景：只留给强情绪、强信息、强压迫

《剑来》项目长期口径：

- 中景为主
- 偏位留白
- 静中有变化
- 高频切镜要少
- 画面服务关系，不追求花哨

### 4.4 站位选择规则

- 单人戏：偏位中景优先
- 双人戏：左右对置或同向半错位
- 同路戏：人物同向，必须有前后半步差
- 多人戏：只保留当前最重要的 2 到 3 人

多人场不要硬塞全员同框。空间连续性优先由背景承担，角色识别优先由当前发言与关系核心承担。

### 4.5 画面构成检查规则

出图前先过三层检查：

1. 空间是否可信
2. 块面是否清楚
3. 风格是否统一

细化标准：

- 先画空间骨架，再画叙事焦点，最后画远景装饰
- 先切大块面，再切中块面，最后补小细节
- 留白必须保留给对白与呼吸，不要恐空乱填
- 人物接触物、受力物、视线所向物的透视优先级最高

如果三者冲突，执行优先级固定为：

1. 空间可信
2. 块面清楚
3. 风格修饰

## 5. 出图提示词的项目固定口径

### 5.1 全项目统一风格词

如无用户明确批准切换风格，默认统一遵守：

- 东方幻想
- 宋代审美
- 半写实
- 山水志怪
- 人间烟火
- 沉静克制
- 电影级构图
- 自然情绪光影
- 高级灰配色
- 真实材质结构

严禁跑偏到：

- 仙侠网游风
- 古偶电视剧风
- 页游风
- 二次元萌系风
- 流行修仙爽文风

### 5.2 图片尺寸

所有新图默认目标尺寸统一为：

`1920 x 1080`

包括：

- 角色立绘
- 背景
- CG
- 场景图
- 宣传图
- 参考图

### 5.3 每次正式出图都要带上的约束包

正式出图默认同时提供：

1. 角色专属身份标准文件
2. 当前唯一母版图
3. 角度辅助图或脸部辅助图
4. 本次允许变化的变量清单
5. 对应场次的分镜与执行口径

推荐固定表述：

`本次不是重新设计角色，而是基于既有母版做同一人物的稳定变体。必须锁定脸型、五官比例、年龄感、发型轮廓、身形体态、气质底色、当前服装结构。本次只允许变化：表情 / 动作 / 场景 / 景别 / 轻微光线。如果新图更好看但不像母版里的同一个人，直接判定失败。`

## 6. 文件命名与资产分级

### 6.1 背景 / 场景图命名

默认格式：

`c章节号_场次号_主体或动作_v版本号`

示例：

- `c1_07_pq_run_v1.png`
- `c1_09_platform_duet_v1.png`
- `c2_02_cpa_pq_v1.png`

要求：

- 全小写
- 只用字母、数字、下划线
- 不用空格
- 不用双下划线
- 文件名尽量短

### 6.2 文档命名

优先短名，推荐：

`主体_用途.md`

示例：

- `cpa_face_ref.md`
- `pq_pose_guide.md`
- `asset_pipeline_sop.md`

### 6.3 角色图片分级命名

默认按资产状态理解，不默认都算正式标准：

- `*_test_*`：测试图
- `*_cand_*`：候选图
- `*_master_*`：正式母版
- `*_exp_sheet_*`：正式表情差分页
- `*_pose_sheet_*`：正式动作页

状态含义固定：

- `test` 不是标准
- `cand` 不是标准
- `master` 才是正式标准

## 7. 什么时候允许回写角色标准文档

出图和改标准文档不是同一个动作。

默认节奏：

1. 先出测试图
2. 再筛候选图
3. 只有正式替换标准时才回写文档

只有下面几类情况才更新角色标准文件或索引：

1. 当前唯一脸部身份母版被替换
2. 当前唯一通用服装母版被替换
3. 默认起手提示词依赖的主母版被替换
4. 正式新增近景锁脸母版并决定以后默认先引用它
5. 正式新增表情差分页并决定以后作为默认参考页
6. 角色标准口径发生实质变化

下面这些情况不要改文档：

- 只是表情更好
- 只是动作更自然
- 只是构图更顺
- 只是生成结果有轻微偏差
- 只是做了一张新场景图
- 只是出现了更好的候选版本但还没确认转正

## 8. 导入项目资源的标准流程

### 8.1 图片导入

新图确认可用后，优先走项目现成脚本：

```powershell
powershell -ExecutionPolicy Bypass -File .\tools\import_generated_image.ps1 -ImageName c1_07_pq_run_v1
```

常见变体：

```powershell
powershell -ExecutionPolicy Bypass -File .\tools\import_generated_image.ps1 -ImageName c2_02_cpa_pq_v1 -Chapter 2
```

```powershell
powershell -ExecutionPolicy Bypass -File .\tools\import_generated_image.ps1 -ImageName c1_08_pq_pose_v1 -SourcePath "C:\path\to\image.png"
```

导入目标：

- `images/chapter1/`
- `images/chapter2/`
- 以此类推

### 8.2 背景定义

图片导入后，在 `bg_images.rpy` 中补：

```renpy
image bg c1_07_pq_run_v1 = im.Scale("images/chapter1/c1_07_pq_run_v1.png", 1920, 1080)
```

然后脚本里只写：

```renpy
scene bg c1_07_pq_run_v1
```

固定要求：

- 文件名 = `image bg` 名 = `scene bg` 调用名
- 若因历史原因保留别名，只能临时兼容
- 主名最终仍要收敛成一个

### 8.3 角色定义

新增说话角色前，先在 `characters.rpy` 补角色变量，再在脚本里使用。

避免：

- 未定义角色直接发言
- 运行时报 `NameError`

### 8.4 配音接入

剧情脚本中统一写：

```renpy
voice voice_id("voice.xxx")
```

不要在剧情里写硬路径。

实际路径统一维护在 `audio_index.rpy` 的 `VOICE_INDEX`。

### 8.5 心境系统接入

涉及情绪节点时，优先使用现成 API：

```renpy
$ story_mood_event("a", "separation")
```

```renpy
$ story_mood_choice("a", "listen")
```

必要时才直接设值：

```renpy
$ set_mood("a", 60, "detail")
```

## 9. Ren'Py 演出落地的标准方法

### 9.1 先按执行表写骨架，不先即兴发挥

每场先写最小可跑骨架：

```renpy
scene bg c2_02_cpa_pq_v1
with dissolve

show cpa standard at left_medium
show peiqian standard at right_medium
with dissolve
```

骨架稳定后，再追加：

- 表情切换
- 站位微调
- 压暗未发言角色
- `voice`
- `music` / `sound`
- `mood` 事件

### 9.2 转场口径

- 日常切入：`with dissolve`
- 压抑停顿：`with fade`
- 奇观或章节收束：`with Dissolve(1.0)`

同场对话中，不要频繁加转场。

### 9.3 多人场口径

- 三人以内固定左中右
- 超过三人只保留当前关键角色
- 其他人由背景空间与对白关系承担，不强行全显

### 9.4 事件图口径

标为 `A3` 的场次优先整图推进，不用普通站姿立绘硬演。

常见方法：

```renpy
hide cpa
hide peiqian
with dissolve

scene bg c2_04_magic_boat_v1
with Dissolve(1.0)
```

即：

- 先清旧立绘
- 再切事件图
- 再进入后续对白或反应镜头

## 10. 每次交付前的静态检查

至少检查以下项目：

### 10.1 脚本检查

- 引号是否闭合
- 字符串是否跨行错误
- `label` / `jump` 是否断链
- 新角色变量是否已定义

### 10.2 资源检查

- 图片路径是否存在
- 音频路径是否存在
- `bg_images.rpy` 是否同步更新
- `scene bg xxx` 是否与 `image bg xxx` 一致
- 是否误写空格、漏字、错版本号、错下划线

### 10.3 演出检查

- 当前场次类型判断是否正确
- 立绘是否误代替事件图
- 近景是否滥用
- 多人场是否塞太满
- 留白是否够对白呼吸
- 当前图是否真的服务这场戏的关系推进

### 10.4 项目口径检查

- 图片尺寸是否为 `1920 x 1080`
- 风格是否仍在项目统一口径内
- 新图是否只是测试 / 候选，而不是误当正式母版

## 11. 验收标准

### 11.1 角色图验收顺序

固定按下面顺序看：

1. 第一眼还是不是那个人
2. 脸部比例有没有漂
3. 年龄感有没有漂
4. 气质底色有没有换人
5. 服装结构有没有跑偏
6. 表情是否真的和上一张不同
7. 动作是否有明确重心、方向和跟随关系

只要前 1 到 5 不稳，就不要因为“更好看”而留。

### 11.2 场景图验收顺序

固定按下面顺序看：

1. 空间是否可信
2. 主块面是否清楚
3. 叙事焦点是否明确
4. 留白是否够对白呼吸
5. 景别是否选对
6. 是否能与已有母图构成统一世界

### 11.3 Ren'Py 落地验收

至少做到：

1. 新游戏从 `label start` 进入不报错
2. 场景切换不黑屏、不卡死
3. 角色名均已定义
4. 图片和音频路径都能正常加载
5. 存档 / 读档 / 快进 / 回看 / 设置可用

---

## 12. 最小执行清单

每次做一张图或一场戏，按这份清单执行：

1. 先确认这次是角色图、动作图还是事件图
2. 先读角色索引、身份标准、总手册、对应分镜表和执行表
3. 写清本次允许变化的变量，禁止直接“重新设计角色”
4. 先做角色稳定，再做表情，再做动作，再做场景
5. 先决定是否复用母图，再决定是否新增图
6. 文件名按项目规则命名，默认 `1920x1080`
7. 导入 `images/chapterX/`
8. 在 `bg_images.rpy` 补 `image bg`
9. 在脚本中只写 `scene bg xxx`
10. 有配音时统一用 `voice_id(...)`
11. 有情绪节点时优先接 `mood` API
12. 做完静态检查，再进入可玩验证

---

## 13. 一句话总结

本项目不是“先画很多图，再拼成 VN”，而是“先锁角色身份，再拆场次演出，再用最少但稳定的资产去承载最多叙事，最后严格按 Ren'Py 资源规范接入工程”。
